# Stage 1.5 addendum — HomeKit control via the `shortcuts` CLI

> Addendum to [`2026-05-29-stage-1-mac-to-lamp.md`](2026-05-29-stage-1-mac-to-lamp.md). Same branch (`stage-1-mac-to-lamp`, PR #3). Execute with subagent-driven-development.

## Why this exists

Stage 1 was built against a Homebridge REST contract assuming the lamp would be bridged via `homebridge-miot`. Discovery with the user corrected that:

- The lamp is a **1st-gen Mi LED Desk Lamp (MJTD01YL)**, already in **Apple Home**, and the user wants to **avoid the Xiaomi cloud**.
- That model has **no native HomeKit** and its **Yeelight LAN control was removed** in firmware — so Homebridge/`homebridge-miot` is the wrong path.
- Confirmed working: drive **Apple Home via the macOS `shortcuts` CLI** (`shortcuts run "Lamp On"` and a fixed-brightness `Lamp 50` both work; the agent's Mac stays logged into the same Apple ID, which Shortcuts requires).

The Stage 1 pipeline (CommandSource → PollLoop → CommandExecutor → ack/stale logic) is unchanged. Only the lamp-control client changes.

## Key constraint: the Home action sets whole state, not single characteristics

Apple's "Control Home" action in Shortcuts captures the accessory's **entire state** (power + brightness + color temperature) as a fixed snapshot — confirmed live: a "set warm" preset also changed brightness. Therefore:

- Single-characteristic presets are impossible. We use **combined presets**, one per (color-bucket × brightness-level) pair.
- The lamp client interface becomes **apply-whole-state**, not three independent setters.

## Combined preset grid (user creates these once)

Default prefix `Lamp`. Grid = 3 color buckets × 3 brightness levels + Off = **10 shortcuts**:

| | 25% | 50% | 100% |
|---|---|---|---|
| **Warm** | `Lamp Warm 25` | `Lamp Warm 50` | `Lamp Warm 100` |
| **Neutral** | `Lamp Neutral 25` | `Lamp Neutral 50` | `Lamp Neutral 100` |
| **Cool** | `Lamp Cool 25` | `Lamp Cool 50` | `Lamp Cool 100` |

Plus `Lamp Off`. Each grid preset sets On + that brightness + that color temperature.

Agent mapping: brightness → nearest of {25, 50, 100}; `color_temp_k` → Warm (≤3300 K) / Neutral (3301–4800) / Cool (>4800). Resolved shortcut name = `Lamp <Bucket> <Level>`.

## Design

**`LampState`** — `{ power: Bool, brightness: Int, colorTempK: Int }`, the fully-resolved desired state.

**`LampClient`** — `{ var apply: @Sendable (LampState) async throws -> Void }`, registered `@Dependency(\.lampClient)`. Two factories:
- `LampClient.shortcuts(prefix:runner:)` — `apply` runs one preset: `!power` → `"<prefix> Off"`; else `"<prefix> <bucket> <level>"` via an injectable `ShortcutRunner`.
- `LampClient.homebridge(baseURL:token:accessoryId:session:)` — retained alternative; `apply` does the power/brightness/colorTemp PUTs (using `miredFromKelvin`). Not the default.

**`ShortcutRunner`** — injectable `@Sendable (_ name: String) async throws -> Void`; live impl runs `/usr/bin/shortcuts run "<name>"` via `Process`, throwing on non-zero exit. Tests inject a recording stub.

**`CommandExecutor`** — resolves a `Command` into a `LampState`, then calls `lampClient.apply`:
- `off` → `power=false`.
- `on` → `power=true`, `brightness = cmd.brightness ?? defaultBrightness (100)`, `colorTempK = cmd.colorTempK ?? defaultColorTempK (2700)`.
- `set` → `power=true` (presets can't change brightness/color without turning on — documented limitation), same brightness/color resolution.

(v1 fills unspecified fields with defaults rather than tracking last-known state — email commands are typically full-intent. Last-state tracking is a noted future enhancement.)

**Config** — `homebridge_*` fields optional; add `lamp_backend` (`"shortcuts"` default | `"homebridge"`) and optional `shortcut_prefix` (default `Lamp`). Required fields: `commands_path`, `poll_interval_s`.
