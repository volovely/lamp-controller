# Stage 1.6 addendum — native HomeKit helper backend

> Addendum to [`2026-05-29-stage-1.5-homekit-shortcuts-addendum.md`](2026-05-29-stage-1.5-homekit-shortcuts-addendum.md). Branch `stage-1-mac-to-lamp`, PR #3.

## OUTCOME (2026-05-31): BLOCKED — not pursued

Task 1.6a (signing feasibility spike) ran and **failed the gate**. `com.apple.developer.homekit` is a restricted capability requiring a provisioning profile that a free/personal team cannot create. Concretely, on this machine (team `3MFN7Y7D69`): `codesign` itself **succeeds** and embeds the entitlement, but at launch the kernel **SIGKILLs the process immediately (exit 137)** because no provisioning profile authorizes the restricted entitlement — it never reaches HomeKit or the "Home access" TCC prompt. (Control: the same binary signed *without* the HomeKit entitlement launches fine.) A profile authorizing HomeKit requires a **paid Apple Developer Program membership ($99/yr)** with the HomeKit capability enabled for the App ID. Secondary friction: `HomeKit` isn't in the native macOS SDK (Mac Catalyst only), so the helper would also need a `macabi` target.

**Decision:** fall back to the Stage 1.5 **Shortcuts backend** (already validated). Tasks 1.6b–1.6e below were NOT executed. This addendum is retained as a record so the helper isn't re-attempted without a paid account. If the user later joins the paid program, 1.6b–1.6e become viable as written.

## Why

The user chose a native HomeKit helper over hand-building ~10 Shortcuts presets. It gives **exact** brightness/Kelvin control (no preset snapping) and offloads the work to code. The lamp is already in Apple Home; the Mac has an "Apple Development" code-signing identity and stays logged into the same Apple ID.

## Risk (drives spike-first ordering)

`com.apple.developer.homekit` is a restricted entitlement. On a **personal** dev team, provisioning a HomeKit-capable profile may not be allowed. If signing fails, **fall back to the Stage 1.5 Shortcuts backend** (already validated working). So Task 1.6a is a feasibility spike; later tasks only proceed if it passes.

## Architecture

A separate signed helper executable (an `.app` bundle for TCC), invoked by the agent — the agent itself stays a plain SPM CLI. New `LampClient.homekit(helperPath:runner:)` backend runs the helper via `Process`, same shape as the Shortcuts backend.

- **`lamp-homekit-helper`** — a minimal macOS app bundle: `Info.plist` with `NSHomeKitUsageDescription`, entitlement `com.apple.developer.homekit`, signed with the dev identity. CLI args set the desired state: `--off` or `--on --brightness <0-100> --color-temp-k <kelvin>`; also `--discover` to print homes/accessories/characteristics (used once to find the accessory + its CT range). Uses `HMHomeManager` (async load via delegate/continuation), finds the configured accessory's `HMService` (lightbulb), writes `On` / `Brightness` / `ColorTemperature` (mired) characteristics, exits 0 on success, non-zero on error.
- **Agent backend** — `LampClient.homekit(helperPath:accessoryName:runner:)`: `apply(state)` invokes the helper with the resolved args. Config: `lamp_backend = "homekit"`, `homekit_helper_path`, `homekit_accessory_name`.
- One-time human steps (cannot be automated): run the signed build, and click **Allow** on the macOS "Home access" prompt on first launch.

- **1.6d — Agent backend + config:** `LampClient.homekit(...)` + `lamp_backend = "homekit"` + config fields + main wiring + tests (stub the helper runner; assert the arg vector for a LampState).

## Fallback
